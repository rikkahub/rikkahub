# Phase 4 announcement-pipeline behavior deltas (PR-2, rikkahub#31)

Deliberate, user-approved changes relative to the pre-Phase-4 double-gated pipeline
(`announcementMutex` + `HermesAnnouncementScheduler.withAnnouncementSlot`). Everything
not listed here is behavior-preserving.

| # | Old behavior | New behavior | Pinned by |
|---|---|---|---|
| 1 | Bridge sends timed out after 30s (`DEFAULT_BRIDGE_SEND_TIMEOUT_MS = 30_000`) because the scheduler's 15s hold ran *inside* the timed send, so the timeout had to paper over the hold. The pre-hack value for every send was 2s. | Sends time out after 5s (`HermesAnnouncer.DEFAULT_BRIDGE_SEND_TIMEOUT_MS = 5_000`); gating happens *before* the send starts, so the timeout no longer has to cover a hold. The same 5s bound governs the manager's tool-response sends too — the queued acknowledgement and the cancel response — not just proactive announcements. | HermesAnnouncerTest `default bridge send timeout is five seconds` (pins the 5_000L constant itself against a silent revert) + HermesJobManagerTest `handleCancelHermesCall records send_timeout when the bridge send exceeds the timeout` (the behavioral timeout path governing a tool-response send — runs with an overridden short `bridgeSendTimeoutMs`, so it does not pin the default value) |
| 2 | The scheduler busy-polled readiness at 200ms ticks (`pollTickMs`) | Event-driven: the quiet-window timer and the blocked-state diagnostic watchdog are re-evaluated on state changes; `pollTickMs` no longer exists. The watchdog records a prolonged block but cannot send an announcement. | HermesAnnouncementLifecycleTest `blocked watchdog diagnoses but never sends` + `quiet window remains an independent safe-boundary blocker`; HermesAnnouncerTest `quiet window delays the send until the timer fires` + `blocked watchdog diagnoses without sending until turn and playback are safe` |
| 3 | An announcement arriving with no bridge attached waited under the mutex until timeout before falling back | Completion/terminal intents fall back to the visible text message immediately; still-working intents drop immediately with `hermes_announcement_dropped_bridge_unavailable` | HermesAnnouncementLifecycleTest `completion intent while detached falls back to text immediately` + `still-working intent while detached is dropped with the unavailable diagnostic, no fallback` |
| 4 | Cross-job announcement order was mutex-acquisition order (unspecified) | Explicit FIFO by enqueue order | HermesAnnouncementLifecycleTest `intents dispatch strictly in enqueue order across jobs` |
| 5 | Announcements were delivered inline in the job actor's effect execution (`awaitJobs()` covered them) | Announcements are async-decoupled: job actors enqueue intents; the announcer's consumer delivers them; `close()` drains queued intents to visible text; the coordinator's drain path awaits announcer quiescence via `awaitClosed()` before scope cancel; process death mid-drain is recovered next session (durable record + replay-on-attach) | HermesAnnouncerTest `awaitClosed drains intents enqueued before close before returning` + `awaitClosed drains tail intents enqueued after close` (drain barrier + tail-after-close regression) are the authoritative quiescence pins — `awaitClosed()` genuinely blocks on those. The coordinator's `stopPersistenceScope()` teardown instead *launches* the drain fire-and-forget into `hermesScope` (`awaitJobs()` → `announcer.awaitClosed()` → store close → scope cancel, all inside that one `launch`); joining would block `endCall` on multi-hour jobs, so it deliberately doesn't. Coordinator-level VoiceAgentRuntimeTest `runtime observability records canonical transcript Hermes and followup attributes` observes delivery only via a bounded poll (`withTimeout(2_000) { while (gemini.textTurns.isEmpty()) delay(10) }`) after `closeAndDrain()`, winning a bounded real-time race today — it does not pin the drain-completes-before-teardown-returns guarantee. |
| 6 | A 15-second maximum-hold threshold could override active assistant audio, and `GenerationComplete` was treated as the pacing boundary. | The 15-second threshold is diagnostic-only. One announcement releases only after Gemini `TurnComplete`, matching playback-epoch-scoped physical drain, and the input quiet window. `Interrupted` flushes playback but never opens the announcement gate. | HermesAnnouncementLifecycleTest gate-matrix/watchdog tests + VoicePlaybackWriterTest ordered drain tests + VoiceAgentPlaybackCoordinationTest interruption regressions |

## Test-move inventory (PR-2, Task 11)

The ten `@Ignore("MOVED-IN-TASK-11")` markers in `VoiceAgentRuntimeTest` were all resolved:
eight runtime journeys were rewritten against the async announcer (routing each assertion
past the point where `HermesAnnouncer` actually delivers the intent), and two were deleted
as pure duplicates of the unit/manager coverage that now owns the behavior.

| Old runtime test | Disposition | Covered by |
|---|---|---|
| `Hermes completion follow up clears prior playback suppression` | Rewritten — wait for the async completion follow-up (which clears suppression) before driving the follow-up audio | itself (audio-playback journey), announcer send by HermesAnnouncerTest `enqueueCompletion sends the completion follow-up and marks the record announced` |
| `Hermes completion follow up audio plays after Gemini interrupts prior response` | Rewritten — wait for the interrupted response's `TurnComplete` and matching physical playback drain before driving follow-up audio | itself (audio-order journey), VoiceAgentPlaybackCoordinationTest `scoped interruption requires the following response completion and drain` |
| `Hermes queued job completion updates history and asks Gemini to explain answer` | Rewritten — wait for the async follow-up turn before asserting on it | itself (history/diagnostics journey), HermesAnnouncerTest `enqueueCompletion sends the completion follow-up and marks the record announced` |
| `Hermes completion announcement waits for assistant audio to stop` | Deleted — focused lifecycle, playback-writer, and coordinator journeys now own the complete safe-boundary contract | HermesAnnouncementLifecycleTest `turn complete alone cannot release while playback is active` + VoicePlaybackWriterTest `turn boundary drains after every previously queued chunk` + VoiceAgentPlaybackCoordinationTest `Hermes completion waits for turn complete and physical playback` |
| `Hermes result completing without a bridge writes a visible chat message once` | Rewritten — wait for the async no-bridge text fallback | itself (visible-message + re-drive journey), HermesAnnouncerTest `detachScoped with no default drains queued intents to text fallback` |
| `Hermes completion follow up failure surfaces answer as visible chat message` | Rewritten — wait for the async failure fallback message | itself (observability/diagnostics journey), HermesAnnouncerTest `completion send returning false appends the visible text fallback` |
| `Hermes long running job gets exactly one still working update` | Deleted — the exactly-once lifecycle is owned at the manager level, while announcer tests drive an explicit turn-plus-drain boundary between progress and completion | HermesJobManagerTest `long running job announces still working once before completion follow-up` + HermesAnnouncerTest `latest progress is followed by final on separate safe boundaries` |
| `Hermes malformed succeeded poll persists failure without leaking answer` | Rewritten — wait for the async terminal follow-up + `announced` flag; the persist/no-leak assertions stay synchronous | itself (persist/no-leak journey), HermesAnnouncerTest `attachScoped replays unannounced terminal results` |
| `Hermes unknown poll status persists failure with job id` | Rewritten — wait for the async terminal follow-up + `announced` flag | itself (persist journey), HermesAnnouncerTest `attachScoped replays unannounced terminal results` |
| `coordinator writes queue e2e events for multiple Hermes jobs` | Rewritten — wait for both async completions and their `late_text_turn_sent` artifact rows, driving a turn-plus-drain boundary between sends | itself (e2e-artifact journey), HermesAnnouncerTest `turn complete plus playback drain pace the second announcement` |
